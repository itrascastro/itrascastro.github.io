1.2. Marc conceptual

1.2.1. Què és un framework de costat client

Un framework és una eina per fer apps web sense recarregar la pàgina. S'executa al navegador i gestiona la interfície de manera dinàmica.

Exemple mínim:

JAVASCRIPT 
// Sense framework: manipular DOM manualment
document.getElementById('llista').innerHTML = '<li>Element 1</li><li>Element 2</li>';

// Amb framework (Angular): declaratiu
<li *ngFor="let item of items"></li>

1.2.2. Per què necessitem frameworks

Problema: Abans havíeu de manipular el DOM manualment per cada canvi. Molt codi repetitiu, difícil de mantenir.

Solució: Els frameworks automatitzen això amb components reutilitzables. Declareu com voleu que es vegi la interfície i ells gestionen les actualitzacions.

1.2.3. Quan usar frameworks

  • Apps amb molta interacció: Filtres, cerca, actualitzacions en temps real
  • Projectes que creixeran: Començar amb estructura clara evita reescriure
  • Equips grans: Convencions comunes faciliten col·laboració
  • Experiència tipus app nativa: Navegació fluida sense recàrregues

1.2.4. Arquitectura client/servidor

En una app moderna:

Servidor:

  • Dades: emmagatzema a BD
  • Lògica: validacions, càlculs
  • Seguretat: autenticació, permisos
  • API: exposa endpoints

Client (amb framework):

  • Presentació: mostra dades amb HTML/CSS
  • Interacció: captura clics, tecles
  • Navegació: canvia vistes sense recarregar
  • Estat local: recorda filtres, seleccions

Aquesta separació permet que backend i frontend treballin en paral·lel.

  • Reactiu: La interfície s'actualitza automàticament quan canvien les dades

1.5. Evolució històrica: des de pàgines estàtiques fins a aplicacions de pàgina única

1.5.1. Objectiu

Entendre com hem passat de pàgines HTML estàtiques a apps web complexes.

1.5.2. Les 4 fases

Fase 1: Pàgines estàtiques (anys 90)

  • HTML escrit a mà
  • Cap interactivitat
  • Cada pàgina = fitxer independent

Fase 2: Pàgines dinàmiques al servidor (finals anys 90 - anys 2000)

  • PHP, ASP generen HTML segons dades
  • JavaScript mínim al client
  • Limitació: cada clic recarrega tot

Fase 3: AJAX (mitjans anys 2000)

  • AJAX permet actualitzar parts sense recarregar
  • jQuery facilita manipular DOM
  • Limitació: codi desorganitzat en créixer

Fase 4: Frameworks moderns (anys 2010 - actualitat)

  • Angular, React, Vue introdueixen components
  • SPA: carreguen tot un cop, navegació local
  • Experiència similar a apps natives

Clau: Els frameworks són la resposta natural a la complexitat creixent de les apps web.

1.6. Què és un marc de treball de costat client

1.6.1. Objectiu

Definir què ofereix un framework.

1.6.2. Definició

Un framework proporciona:

  • Components: peces reutilitzables (botó, llista, formulari)
  • Plantilles: sintaxi per descriure HTML dinàmic
  • Estat: sincronitza dades entre components
  • Rutes: navegació sense recarregar
  • CLI: eines per crear projectes i desenvolupar

1.6.3. Framework vs Llibreria

  • Llibreria: Crideu funcions quan vulgueu (jQuery)
  • Framework: Defineix l'estructura i crida el vostre codi (inversió de control)

1.6.4. Funcions principals

  1. Renderitzar interfície: Components generen HTML dinàmic
  2. Gestionar estat: Mantenir dades sincronitzades
  3. Enrutament: URLs sense recarregar (/productes, /carret)
  4. Serveis HTTP: Comunicar amb API del servidor

Clau: Framework = estructura + eines + ecosistema.

1.6.5. Resultat esperat

1.7. Comparativa de frameworks populars

1.7.1. Objectiu

Conèixer les opcions principals del mercat.

1.7.2. Els 4 grans

Angular (Google, 2010)

  • Complet: tot inclòs (rutes, HTTP, formularis)
  • TypeScript obligatori
  • Opinió forta: una manera correcta de fer-ho
  • Ideal per: projectes grans, equips, educació

React (Meta/Facebook, 2013)

  • Llibreria enfocada en la vista
  • JSX: HTML dins JavaScript
  • Flexible: trieu llibreries externes
  • Ideal per: projectes de tot tipus, més popular

Vue (comunitat, 2014)

  • Progressiu: useu el que necessiteu
  • Sintaxi clara i fàcil d'aprendre
  • Balanç entre Angular i React
  • Ideal per: prototips ràpids, equips petits

Svelte (comunitat, 2016)

  • Compila a JavaScript optimitzat
  • Sense runtime virtual DOM
  • Menys codi, més rendiment
  • Ideal per: apps amb restriccions de pes

1.7.3. Per què triem Angular al curs?

  • Estructura clara: Aprendre patrons des del principi
  • Tot inclòs: No cal triar 10 llibreries externes
  • TypeScript: Detecció d'errors abans d'executar
  • Estabilitat: Versions predictibles, suport Google

Clau: Angular és ideal per aprendre fonaments abans d'explorar altres opcions.

1.8. Avantatges dels frameworks moderns

1.8.1. Objectiu

Entendre per què els frameworks milloren la productivitat.

1.8.2. Avantatges principals

1. Productivitat

  • Generadors de codi: ng generate component
  • Hot reload: veieu canvis a l'instant
  • CLI integrat: un comando per tot

2. Mantenibilitat

  • Estructura clara: components, serveis, rutes
  • Convencions: tots segueixen el mateix patró
  • Testing integrat: proves automàtiques

3. Ecosistema

  • Llibreries: milers de components fets
  • Documentació: guies oficials extenses
  • Comunitat: solucions a Stack Overflow

4. Rendiment

  • Virtual DOM: actualitza només el mínim necessari
  • Lazy loading: carrega codi sota demanda
  • Optimitzacions automàtiques

Clau: Framework = menys temps resolent problemes comuns, més temps creant valor.

1.9. Components clau d'un framework

1.9.1. Objectiu

Identificar les peces fonamentals.

1.9.2. Els 4 pilars

1. Components

Peces reutilitzables amb lògica pròpia.

TYPESCRIPT 
@Component({
  selector: 'app-targeta',
  template: '<div></div>'
})

2. Rutes

Navegació sense recarregar.

TYPESCRIPT 
{path: 'cataleg', component: CatalegComponent}

3. Serveis

Lògica compartida entre components.

TYPESCRIPT 
catalogService.getElements()

4. Plantilles

HTML dinàmic amb sintaxi especial.

HTML 
<li *ngFor="let item of items"></li>

Clau: Components (vista) + Serveis (lògica) + Rutes (navegació) = app completa.

1.10. Casos d'ús reals: quan utilitzar frameworks

1. Pàgines informatives

  • Exemple: Pàgina presentació producte
  • Raó: JavaScript mínim = càrrega ràpida

3. Restriccions de rendiment

  • Exemple: Dispositius antics
  • Raó: Frameworks afegeixen pes

4. Prototips ràpids sense experiència

  • Exemple: Prototip en 2 dies

2.2. Marc conceptual

2.2.1. Què necessitem per desenvolupar

Abans d'instal·lar, cal entendre l'ecosistema complet. Desenvolupar amb Angular requereix 7 capes d'eines que treballen juntes.

2.2.2. Les 7 peces de l'ecosistema

  1. Sistema operatiu: Windows, macOS o Linux amb recursos suficients
  2. Node.js: Motor per executar JavaScript fora del navegador
  3. npm: Gestor de paquets per instal·lar llibreries
  4. Angular CLI: Comandes per crear projectes i components
  5. Editor: VS Code amb extensions específiques
  6. Git: Control de versions per guardar canvis
  7. Navegador: Chrome/Firefox amb DevTools per depurar

2.2.3. Per què tantes eines

Cada eina resol un problema específic:

Node.js: Executa el servidor de desenvolupament i compila el codi.

BASH 
# Node.js permet executar comandes com:
ng serve  # Llença servidor local

npm: Gestiona dependències automàticament.

BASH 
# Instal·la 1.000+ paquets en segons
npm install

Angular CLI: Genera codi amb convencions.

BASH 
# Crea component amb HTML, CSS, TypeScript i test
ng generate component targeta

VS Code: Autocompletat i detecció d'errors en temps real.

Git: Permet desfer canvis i treballar en equip sense perdre feina.

2.2.4. Quan necessitem cada eina

  • Node.js + npm: SEMPRE (base de tot)
  • Angular CLI: Per crear i gestionar projectes Angular
  • VS Code: Recomanat (alternativa: WebStorm, Sublime)
  • Git: Imprescindible per projectes professionals
  • DevTools: Depurar l'app al navegador

2.3.2. Conceptes clau

  • PATH: Variable del sistema que indica on buscar executables. Si no està configurat, les comandes no funcionen.

2.5. Requisits de maquinari: RAM, processador, espai en disc

2.5.1. Objectiu

Saber si el vostre equip té prou recursos.

2.5.2. Taula de requisits

Component Mínim Recomanat
CPU 2 nuclis, 2 GHz 4+ nuclis, 3+ GHz
RAM 4 GB 8-16 GB
Disc 10 GB lliures (HDD) 20+ GB lliures (SSD)

2.5.3. Per què aquests números

  • CPU: Compilacions TypeScript + servidor ng serve = ús intensiu. Més nuclis = més ràpid.
  • RAM: VS Code + navegador + servidor = 3-4 GB simultàniament. Amb 8 GB treballa còmode.
  • SSD: npm instal·la milers de fitxers petits. SSD accelera 3-5x vs HDD.

2.5.4. Com comprovar l'equip

Windows:

POWERSHELL 
systeminfo | findstr /C:"Processor" /C:"Total Physical Memory"

macOS:

BASH 
sysctl -n machdep.cpu.brand_string
sysctl hw.memsize

Linux:

BASH 
lscpu | grep "Model name"
free -h

2.5.5. Conseqüències de no complir mínims

  • Poca RAM: Swap (memòria virtual) alenteix tot
  • CPU lenta: Compilacions de 5s passen a 60s
  • Poc espai: No podreu instal·lar dependències

Clau: 8 GB RAM + SSD = experiència fluida.

3.11. Resolució de problemes comuns

3.11.1. Objectiu

Diagnosticar i resoldre errors típics.

3.11.2. Problema 1: "command not found: node"

Causa: PATH no actualitzat o Node.js no instal·lat.

Solució:

BASH 
# 1. Tancar i reobrir terminal
# 2. Verificar PATH
echo $PATH  # macOS/Linux
$env:PATH   # Windows

# 3. Si no hi és, afegir manualment
# Windows: Editar variables d'entorn del sistema
# macOS/Linux: Afegir a ~/.bashrc o ~/.zshrc
export PATH="/usr/local/bin:$PATH"

3.11.3. Problema 2: "npm ERR! code EACCES"

Causa: Permisos insuficients.

Solució Linux/macOS:

BASH 
# Configurar prefix npm
mkdir ~/.npm-global
npm config set prefix '~/.npm-global'

# Afegir al PATH
echo 'export PATH=~/.npm-global/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

Solució Windows:

Executar PowerShell com administrador per instal·lacions globals.

3.11.4. Problema 3: Instal·lacions lentes

Causa: Caché corrupta.

Solució:

BASH 
npm cache clean --force
npm cache verify
npm install

3.11.5. Problema 4: npm WARN deprecated

Causa: Paquet obsolet.

Interpretació:

  • WARN no és error - paquet instal·lat correctament
  • Només avís que el paquet ja no es manté
  • Podeu ignorar tret que hi hagi problemes funcionals

3.11.6. Problema 5: npm ERR! network

Causa: Problemes xarxa o proxy.

Solució:

BASH 
# Verificar connexió
npm ping

# Configurar proxy (si escau)
npm config set proxy http://proxy.example.com:8080
npm config set https-proxy http://proxy.example.com:8080

Clau: La majoria de problemes es resolen reiniciant terminal o netejant caché.

3.18. Repàs de conceptes
  • Node.js: motor JavaScript basat en V8 per executar codi fora del navegador.
  • npm: gestor de paquets que s'instal·la automàticament amb Node.js.
  • LTS: versions estables mantingudes 30 mesos; recomanades per producció.
  • PATH: variable del sistema per buscar executables; necessària per node i npm.
  • package.json: manifest del projecte amb dependències.
  • node_modules: carpeta amb paquets instal·lats localment.
  • npm init: crear nou projecte amb package.json.
  • npm install: descarregar i instal·lar paquets.
  • npm run: executar scripts de package.json.
