1.2. Marc conceptual

1.2.1. Què és un IDE

Un IDE (Integrated Development Environment) és una aplicació que integra múltiples eines en un sol lloc: editor amb ressaltat de sintaxi, depurador, terminal, control de versions i extensions.

1.2.2. Per què necessitem un IDE

Problema: Escriure codi amb editor de text pla (Notepad, TextEdit) no detecta errors, no suggereix completacions, no integra Git, no executa comandes.

Solució: Un IDE modern com VS Code ofereix:

  • Autocompletat intel·ligent
  • Detecció d'errors en temps real
  • Terminal integrada
  • Control de versions (Git)
  • Extensions per frameworks específics

1.2.3. Exemple: Editor pla vs VS Code

Editor de text pla:

TYPESCRIPT 
// Escriviu tot manualment, cap ajuda
const nom = "Joan"
console.log(nom.toUperCase())  // Error! però no el detecta

VS Code amb extensions:

TYPESCRIPT 
// VS Code detecta l'error i suggereix
const nom = "Joan"
console.log(nom.toUpperCase())  // Autocompletat, error subratllat
//              ^^^^^^^ ESLint: typo detectat!

1.2.4. Quan usar un IDE complet vs editor lleuger

IDE complet (WebStorm, IntelliJ):

  • Projectes grans empresarials
  • Equips amb pressupost
  • Refactorització complexa
  • Depuració avançada

Editor extensible (VS Code):

  • Projectes de tot tipus (petits a grans)
  • Gratuït i de codi obert
  • Lleuger i ràpid
  • Extensible amb plugins
  • Recomanat per aquest curs

1.2.5. Per què VS Code per aquest mòdul

Avantatges:

  • Gratuït i de codi obert
  • Multiplataforma: Windows, macOS, Linux
  • Lleuger: Arrenca en 2-3 segons
  • Suport natiu TypeScript: Creat per Microsoft
  • Extensions: Milers al Marketplace
  • Terminal integrada: Executa npm, ng, git sense sortir
  • IntelliSense: Autocompletat intel·ligent
  • Git integrat: Control de versions visual

1.3. Model mental

1.3.1. Arquitectura de VS Code

TEXT 
VS Code (nucli lleuger)
    ↓
Extensions (capes funcionals)
    ├─ Angular Language Service → Suport Angular
    ├─ ESLint → Qualitat codi
    ├─ Prettier → Format automàtic
    ├─ GitLens → Git visual
    └─ Live Server → Servidor HTTP
    ↓
Terminal integrada (pont amb sistema)
    ├─ npm (gestió paquets)
    ├─ ng (Angular CLI)
    └─ git (control versions)
    ↓
Sistema operatiu (Windows/macOS/Linux)

1.3.2. Conceptes clau

  • Editor com a nucli: VS Code és el punt central on escrivim codi, obrim fitxers i gestionem el projecte
  • Extensions com a capes: Afegeixen funcionalitats específiques sense sobrecarregar l'aplicació base
  • Terminal integrada com a pont: Permet executar npm, ng, git sense sortir de l'editor
  • Configuració multiplataforma: Les mateixes extensions funcionen a Windows, macOS i Linux
  • PATH global: Comanda code accessible des de qualsevol terminal per obrir fitxers/carpetes
  • IntelliSense: Sistema d'autocompletat que entén el context del codi TypeScript/JavaScript
  • Format On Save: Prettier s'executa automàticament en desar (Cmd/Ctrl+S)

1.3.4. Àrees de la interfície

  • Barra activitats (esquerra): Explorer, Search, Source Control, Run, Extensions
  • Barra lateral: Contingut de la vista activa (fitxers, cerca, Git...)

1.4. Què és un IDE i diferències amb un editor de text pla

1.4.1. Objectiu

Entendre les diferències fonamentals entre un editor de text pla i un IDE modern.

1.4.2. Editor de text pla vs IDE

Editor de text pla (Notepad, TextEdit, gedit):

  • Només edita text sense format
  • No detecta errors de sintaxi
  • No suggereix completacions
  • No integra eines de depuració
  • No entén estructura de projectes

1.4.3. Taula comparativa

Funció Editor pla IDE modern
Sintaxi destacada No Sí
Autocompletat No Sí
Errors en temps real No Sí
Terminal integrada No Sí
Control versions No Sí (Git)
Extensions No Sí (Marketplace)
Depurador No Sí (breakpoints)
Refactorització No Sí (F2)

1.4.4. Per què és essencial

Desenvolupar Angular sense IDE:

  • Escriure tot manualment, propens a errors
  • Canviar entre 10 finestres (editor, terminal, navegador, Git...)
  • No detectar errors fins executar
  • Molt més lent i menys productiu

Desenvolupar Angular amb VS Code:

  • Tot en un lloc
  • Errors detectats abans d'executar
  • Autocompletat accelera escriptura
  • Terminal integrada
  • Git visual

Clau: IDE integra múltiples eines en un sol lloc, multiplicant la productivitat.

1.6. Descàrrega i instal·lació de Visual Studio Code

1.6.1. Objectiu

1.6.2. Pas 1: Descarregar l'instal·lador

Aneu a https://code.visualstudio.com

  • Windows: VSCodeUserSetup-x64-1.XX.X.exe (~80 MB)
  • macOS: VSCode-darwin-universal.zip (~100 MB)
  • Linux: .deb (Ubuntu/Debian) o .rpm (Fedora)

1.6.3. Pas 2: Instal·lar a Windows

  1. Executar instal·lador
  2. Acceptar llicència
  3. IMPORTANT: Marcar:
    • Add "Open with Code" to context menu
    • Add to PATH (enables 'code' command)
    • Register Code as editor for supported files
  4. Install → Finish
  5. VS Code s'obre automàticament

1.6.4. Pas 3: Instal·lar a macOS

Opció 1: Instal·lador gràfic

  1. Descomprimir .zip
  2. Arrossegar Visual Studio Code.app a Applications
  3. Obrir des de Launchpad

Opció 2: Homebrew

BASH 
brew install --cask visual-studio-code

Afegir comanda code al PATH:

  1. Obrir VS Code
  2. Cmd+Shift+P → Shell Command: Install 'code' command in PATH
  3. Reiniciar terminal

1.6.5. Pas 4: Instal·lar a Linux

Ubuntu/Debian:

BASH 
wget -O vscode.deb https://code.visualstudio.com/sha/download?build=stable&os=linux-deb-x64
sudo apt install ./vscode.deb

Fedora:

BASH 
sudo rpm --import https://packages.microsoft.com/keys/microsoft.asc
sudo dnf install code

BASH

1.9. Extensions imprescindibles

1.9.1. Objectiu

Entendre quines extensions necessitem i per què.

1.9.2. Les 5 extensions essencials

1. Angular Language Service:

  • Autocompletat en plantilles HTML Angular
  • Errors en temps real a plantilles
  • Navegació definicions (F12)

2. ESLint:

  • Analitza qualitat codi TypeScript/JavaScript
  • Detecta errors, estil inconsistent
  • Configurable amb .eslintrc.json

3. Prettier:

  • Format automàtic consistent
  • Aplica regles d'estil (cometes, punts i comes...)
  • Integra amb "Format On Save"

4. GitLens:

  • Millora integració Git
  • Mostra autor de cada línia (blame)
  • Historial de canvis visual

5. Live Server (opcional):

  • Servidor HTTP local per HTML estàtic
  • Recàrrega automàtica en desar
  • Útil per provar HTML/CSS fora d'Angular

1.9.3. Per què aquestes extensions

Sense extensions:

  • Errors no detectats fins executar
  • Format manual inconsistent
  • No autocompletat en plantilles Angular
  • Git només per terminal

Amb extensions:

  • Errors detectats en escriure
  • Format automàtic en desar (Cmd/Ctrl+S)
  • Autocompletat intel·ligent
  • Git visual i intuïtiu
Informació
Clau: Extensions converteixen VS Code en IDE específic per Angular.
Èxit
Consell: Instal·leu només les extensions necessàries. Massa extensions poden alentir VS Code. Comenceu amb les 5 essencials i afegiu més només si necessiteu funcionalitats específiques.

2.2. Marc conceptual

2.2.1. Què és el control de versions

El control de versions és un sistema que registra canvis sobre fitxers al llarg del temps, permetent recuperar versions específiques quan calgui.

2.2.2. Per què necessitem control de versions

Problema sense control de versions:

Caos, pèrdua de codi, conflictes, manca d'historial

Solució amb Git:

TEXT 
projecte/

Un sol directori, tots els canvis guardats, recuperació fàcil

2.2.3. Problemes que resol Git

1. Pèrdua de codi:

  • Sense Git: Esborrar fitxer = perdut per sempre
  • Amb Git: git checkout recupera qualsevol versió anterior

2. Conflictes de col·laboració:

  • Sense Git: Dues persones editen → canvis sobreescrits
  • Amb Git: Fusió automàtica o detecció de conflictes

3. Manca d'historial:

  • Sense Git: Qui va fer què? Per què? Quan?
  • Amb Git: git log mostra autor, data, missatge de cada canvi

4. Por a experimentar:

  • Sense Git: No podeu provar coses noves, podríeu trencar tot
  • Amb Git: Crea branca, experimenta, si no funciona → elimina branca

2.2.4. Què és Git

Git és un sistema de control de versions distribuït creat per Linus Torvalds (2005) per gestionar el nucli Linux.

Característiques:

  • Ràpid: Operacions locals, sense xarxa
  • Distribuït: Cada ordinador té l'historial complet
  • Branques lleugeres: Crear branca = instant, sense espai extra
  • Integritat: Hash SHA-1 garanteix no corrupció
  • Estàndard: Linux, VS Code, Angular, React, milions de projectes

2.3. Model mental

2.3.1. Tres zones de Git

TEXT 
┌─────────────────────┐
│ Directori de treball │  ← Editeu fitxers aquí
│   (Working Directory)│
└──────────┬───────────┘
           │ git add
           ↓
┌─────────────────────┐
│  Àrea d'stage       │  ← Sala d'espera
│  (Staging Area)     │
└──────────┬───────────┘
           │ git commit
           ↓
┌─────────────────────┐
│   Repositori        │  ← Historial permanent
│   (Repository)      │
└─────────────────────┘

2.3.2. Flux de treball bàsic

TEXT 
[Editar fitxer] → git add → [Stage] → git commit → [Repositori]
       ↑                                                |
       └─────────────── git checkout ─────────────────┘

2.3.3. Conceptes clau

  • Directori de treball: Carpeta visible amb fitxers editables. Canvis aquí no estan guardats a Git.
  • Stage (Staging Area): Zona intermèdia. Fitxers aquí estan preparats per commit.
  • Repositori: Carpeta .git/ amb tot l'historial. Commits aquí són permanents.
  • Commit: Instantània immutable amb autor, data, missatge i hash únic (SHA-1).
  • Branch: Línia de desenvolupament. Defecte: main (abans master).
  • HEAD: Punter indicant commit/branca actual.
  • Distribuït: Cada ordinador té còpia completa de l'historial. No depèn de servidor central.

2.3.4. Metàfora: Git com a màquina del temps

Git és una màquina del temps per al codi:

  • Cada commit = foto del projecte en un moment
  • git log = àlbum de fotos cronològic
  • git checkout = viatjar a una foto anterior
  • Branca = línia temporal alternativa (multiversos!)

2.3.5. Exemple pràctic pas a pas

BASH 
# 1. Crear repositori
mkdir projecte
cd projecte
git init  # Crea .git/

# 2. Editar fitxer (directori de treball)
echo "Hello" > hello.txt
git status  # hello.txt = untracked

# 3. Afegir a stage
git add hello.txt
git status  # hello.txt = staged (verd)

# 4. Commit (repositori)
git commit -m "Afegit hello.txt"
git status  # Nothing to commit (net)

# 5. Consultar historial
git log --oneline  # Mostra commit amb hash i missatge

2.3.6. Errors comuns a evitar

  • Oblidar git add abans de git commit: Commit buit, res es guarda
  • Missatges de commit vagues: "canvis" vs "Afegida validació email"
  • No fer commits freqüents: Esperar 2 dies = perds historial detallat
  • Fer commits massa grans: 100 fitxers en 1 commit = difícil revisar

Aquest model mental us ajuda a entendre per què Git és tan potent per gestionar canvis.

2.5.1. Objectiu

2.5.2. Conceptes fonamentals

2.17. Repàs de conceptes
  • Control de versions: sistema que registra canvis sobre fitxers al llarg del temps, permetent recuperar versions específiques.
  • Git: sistema de control de versions distribuït, ràpid i usat per milions de projectes.
  • Repositori: contenidor que emmagatzema projecte i tot el seu historial. Carpeta .git.
  • Commit: instantània immutable del projecte amb autor, data, missatge i hash únic (SHA-1).
  • Stage (Staging Area): zona intermèdia on es preparen canvis abans de confirmar.
  • Historial: seqüència ordenada de commits que mostra evolució del projecte.
  • Branca (branch): línia de desenvolupament independent. Defecte: main.
  • Comandes essencials:
  • git init: inicialitza repositori nou.
  • git status: mostra estat actual (modificats, staged, untracked).
  • git add <fitxer>: afegeix fitxers a stage.
  • git add .: afegeix tots els fitxers.
  • git commit -m "missatge": crea commit amb canvis de stage.
  • git commit -am "missatge": afegeix i confirma fitxers ja seguits en una comanda.
  • git log: mostra historial complet.
  • git log --oneline: historial format compacte.
  • Configuració global:
  • git config --global user.name "Nom": defineix nom autor.
  • git config --global user.email "email": defineix correu autor.
  • git config --global core.editor "code --wait": defineix VS Code com a editor.
  • git config --global init.defaultBranch main: defineix main com a branca per defecte.
  • Source Control (VS Code): interfície gràfica per gestionar Git visualment (stage, commit, diff).

3.2. Marc conceptual

3.2.1. Què és un repositori remot

Un repositori remot és una còpia del repositori emmagatzemada en un servidor accessible per Internet. Funciona com "còpia de seguretat intel·ligent" que també facilita col·laboració.

3.2.2. Problema: només repositoris locals

Situació sense remots:

TEXT 
Ordinador local:
~/workspace/projecte/.git  ← Tot l'historial aquí

Problemes:

  • Disc dur es trenca → tot perdut
  • Vols treballar des de casa → has de portar l'ordinador
  • Col·laborar amb altres → copiar fitxers manualment (caos)
  • Cap còpia de seguretat automàtica

3.2.3. Solució: repositori remot

Situació amb remots:

TEXT 
Ordinador local:          Internet:
~/workspace/projecte/.git ←→ GitHub: github.com/usuari/projecte

Avantatges:

  • Còpia de seguretat automàtica (cada push)
  • Treballa des de múltiples dispositius (casa, universitat, portàtil)
  • Col·laboració en equip (múltiples persones)
  • Revisió de codi (pull requests)
  • Portafoli professional (mostra projectes)
  • Integració contínua (CI/CD automàtic)

3.2.4. Exemple real: recuperació després de fallada

Sense remot:

BASH 
# Disc dur es trenca
→ Tot perdut per sempre
→ Has de refer 3 mesos de feina

Amb remot:

BASH 
# Disc dur es trenca
→ Compres ordinador nou
→ git clone https://github.com/usuari/projecte
→ Tot recuperat en 2 minuts!

3.3.3. Conceptes clau

  • Local: Repositori al vostre ordinador. Treballeu aquí (edit, add, commit).
  • Remot: Repositori al servidor (GitHub). Còpia de seguretat i sincronització.

  • Pull: Baixar commits remot → local. Fusiona automàticament.
  • Clone: Crear còpia local completa d'un remot. Inclou tot l'historial.
  • Fork: Còpia d'un repositori remot al vostre compte GitHub. Per contribuir a projectes d'altres.

Git + GitHub:

  • Repositori sincronitzat manualment (git push/pull)

3.4. Repositori local vs remot: diferències i avantatges

3.4.1. Objectiu

Entendre diferències entre local i remot, i per què necessitem ambdós.

3.4.2. Repositori local

Característiques:

  • Ubicació: Carpeta amb .git al disc local
  • Accés: Només des del vostre ordinador
  • Velocitat: Molt ràpid (operacions locals)
  • Internet: No necessari
  • Risc: Vulnerable a pèrdues (disc, robatori, virus)

Operacions locals:

BASH 
git init, git add, git commit, git log, git branch

Avantatges:

  • Ràpid (sense latència xarxa)
  • Funciona offline
  • Privat (ningú veu canvis)

Desavantatges:

  • Pèrdua de dades si disc falla
  • No pots col·laborar
  • Un sol dispositiu

3.4.3. Repositori remot

Característiques:

  • Ubicació: Servidor GitHub al núvol
  • Accés: Des de qualsevol ordinador amb Internet
  • Velocitat: Depèn de la xarxa
  • Internet: Imprescindible
  • Risc: Protegit per GitHub (còpies seguretat)

Operacions remotes:

BASH 
git push, git pull, git clone, git fetch

Avantatges:

  • Còpia de seguretat automàtica
  • Col·laboració fàcil
  • Múltiples dispositius
  • Portafoli professional

Desavantatges:

  • Necessita Internet
  • Latència (més lent que local)
  • Públic si no pagat (gratis = públic o limitat privat)

3.4.4. Per què necessitem ambdós

Local: On treballeu (edit, add, commit)

Remot: On deseu i compartiu (push, pull, clone)

Analogia:

  • Local = quadern personal (treballes aquí)
  • Remot = còpia al núvol (protegeixes i comparteixes)
Informació
Clau: Local per treballar ràpid offline; remot per protegir i col·laborar.

3.11. Concepte de clonació

3.11.1. Objectiu

Entendre git clone per crear còpies locals.

3.11.2. Què és clonar

Clonar = Crear còpia local completa d'un repositori remot.

3.11.3. Per què clonar

Casos d'ús:

  • Treballar des d'altre ordinador
  • Col·laborar en projecte d'altri
  • Recuperar projecte després de perdre local

3.11.4. Comanda bàsica

BASH 
git clone <URL>

Exemple:

BASH 
cd ~/workspace/altres-projectes
git clone https://github.com/nom-usuari/primer-repo-remot.git
cd primer-repo-remot

3.11.5. Què inclou un clone

Clone crea:

  • Carpeta amb nom del repositori
  • Tot l'historial de commits
  • Configuració remot origin
  • Branca main (o la per defecte)

Verificar:

BASH 
git log --oneline  # Tot l'historial
git remote -v      # Origin ja configurat

3.11.6. Clone vs descarregar ZIP

git clone Descarregar ZIP
Tot l'historial Només fitxers actuals
Pot fer push/pull No té .git
Connexió amb remot Cap connexió
Treballa amb branques Versió estàtica

3.11.7. Exemple: treballar des de 2 ordinadors

Ordinador casa:

BASH 
git clone https://github.com/nom-usuari/projecte.git
# Treballar, commit, push

Ordinador universitat:

BASH 
git clone https://github.com/nom-usuari/projecte.git
git pull  # Baixar canvis de casa
# Treballar, commit, push

Ordinador casa (dia següent):

BASH 
git pull  # Baixar canvis de la universitat
Informació
Clau: git clone crea còpia local completa; git pull sincronitza canvis posteriors.

3.11.8. Resultat esperat

Compreneu git clone i podeu crear còpies locals de repositoris remots.

3.17. Repàs de conceptes
  • Repositori local: Còpia del projecte amb historial al nostre ordinador (carpeta amb .git).
  • Repositori remot: Còpia del projecte al servidor accessible per Internet (GitHub, GitLab, Bitbucket).
  • GitHub: Plataforma d'allotjament de repositoris Git més popular, propietat de Microsoft.
  • Origin: Nom per defecte del repositori remot principal.
  • Comandes remotes:
    • git remote add origin <URL>: Connecta repositori local amb remot.
    • git remote -v: Llista remots configurats.
    • git push -u origin main: Publica commits locals al remot primera vegada (configura tracking).
    • git push: Publica commits locals al remot (després de tracking).
    • git pull origin main: Baixa commits del remot i fusiona amb local.
    • git clone <URL>: Crea còpia local completa d'un repositori remot.
  • Personal Access Token (PAT): Token d'autenticació per accedir GitHub des de terminal (substitueix contrasenya).
  • Repositori públic: Visible per tothom, ideal per open source i portfolis.
  • Repositori privat: Només visible per vós i convidats, ideal per projectes confidencials.
  • Tracking: Relació entre branca local i remota que permet git push sense especificar remot.
  • Upstream: Branca remota de la qual fa seguiment una branca local.
  • Fork: Còpia d'un repositori remot al vostre compte GitHub (per contribuir a projectes d'altres).
  • Clonació: Crear còpia local completa d'un repositori remot, incloent tot l'historial.
